# Hermes Prompt — Build and Enhance the Laminate Changer Tool

Use this prompt inside Hermes when you want it to implement and harden the laminate changer tool.

```text
Build a premium, production-grade Laminate Changer tool for the design/render vertical.

GOAL
Allow users to change laminate, veneer, shutter, panel, and cladding finishes on a rendered interior scene while preserving geometry, lighting, edges, hardware, seams, and room style.

THIS TOOL MUST BE BETTER THAN A BASIC INPAINT TOOL.
It must understand:
- surface category
- material finish
- grain direction
- seam continuity
- lighting/reflection realism
- style consistency with the room
- preservation of hardware and neighboring materials

SUPPORTED SURFACE TYPES
- wardrobe_shutter
- cabinet_front
- tv_panel
- wall_panel
- vanity_shutter
- partition_cladding
- headboard_panel
- shelf_panel
- laminate_floor (advanced)

SELECTION MODES
- auto_detect
- detected_surface
- manual_mask
- lasso
- rectangle

ARCHITECTURE
Use the following layers:
1. frontend workspace / premium UI
2. API/domain service
3. worker job processor
4. AURA edit planner
5. provider/inference gateway image edit execution
6. output validation + critique
7. render history persistence

REQUIRED FILES / MODULES
- laminate-changer.service.ts
- laminate-changer.processor.ts
- laminate-changer.api.ts
- laminate-changer.types.ts
- laminate-changer.schemas.ts
- laminate-changer.prompts.ts
- useLaminateChanger.ts
- LaminateChangerWorkspace.tsx
- SurfaceCandidateOverlay.tsx

AURA REQUIREMENTS
Use AURA for:
- surface-aware edit planning
- material change instructions
- must_keep / must_avoid generation
- room-consistency notes
- critique after output

Do NOT use AURA for exact geometry extraction.

PIPELINE
1. load render + project + zone context
2. detect candidate surfaces or accept manual mask
3. resolve selected material reference from catalog or free text
4. generate AURA structured laminate change plan
5. run image edit through inference gateway
6. validate that change stayed within intended area
7. critique realism and finish believability
8. save result as child render
9. store lineage, metadata, and critique

VALIDATION / SAFETY
- if auto surface detection confidence is low, require manual confirmation
- if edit spills into adjacent geometry, fail validation
- if critique fails badly, mark as low-confidence output and recommend retry
- preserve hardware if requested
- preserve panel seams if requested
- preserve reflections/shadows if requested

UI / UX DIRECTION
This tool should feel like a premium material-editing studio, not a plain form.

The workspace should include:
- large hero render preview
- surface overlays
- high-end right-side control panel
- material reference input/search
- style intent notes
- execution notes
- preview plan generated by AURA
- clear Apply CTA

The UI should feel:
- dark, premium, calm, architectural
- visually stronger than typical SaaS tools
- presentable to interior firms and enterprise clients

DATA / VERSIONING
- every laminate change creates a child render/version
- preserve parent render lineage
- store request + plan + validation + critique in metadata
- support future compare/history features

DONE MEANS DONE
The tool is only done when:
- the request path is validated with schemas
- the worker path is async and durable
- the AURA plan is structured JSON
- the output is validated and critiqued
- the result is stored as a versioned child render
- the UI is premium and coherent with the rest of the platform
- failure states and manual fallback paths are clean

Use the scaffold files already provided if they exist, but improve and integrate them into the real repo structure.
```
